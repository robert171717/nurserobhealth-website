# Content Scheduler v1.0

Auto-pushes Nurse Rob content to X at 9AM and 5PM MST via xurl CLI with validation, logging, and cross-posting.

## Optimal Posting Schedule (MST)
| Slot | Time | Content Type |
|------|------|-------------|
| Morning | 9:00 AM | Educational Thread/Carousel |
| Evening | 5:00 PM | Engagement/Poll/Myth or Short Form/Insight |

## Workflow

### Step 0: Pre-Flight — Verify xurl Auth
**BEFORE attempting any post**, verify xurl is authenticated:
```bash
xurl auth status
```
Expected: at least one registered app with an active oauth2 token and a default app set (marked with `▸`).

**State A — No apps registered or no token:** Output shows `No apps registered` or the default app has `oauth2: (none)`.
- **DO NOT attempt to post** — every post call will fail.
- Log the failure to `post_log.json` with status `FAILED` and error `xurl auth not configured`.
- Surface setup instructions in final response.
- **Still proceed with Step 1 (load/generate content) and Step 2 (validate)** — content should be generated and validated even when auth is blocked, so it's ready immediately when auth gets fixed. Only skip posting steps (Steps 3-6). Then go to Step 7 with the gap flagged and note that content is ready-and-waiting.

State A fix (first-time setup — user must complete outside agent):
```bash
xurl auth apps add my-app --client-id ID --client-secret SECRET
xurl auth oauth2 --app my-app YOUR_USERNAME
xurl auth default my-app
```
Credentials from https://developer.x.com/en/portal/dashboard (min $5 API credits).

---

**State B — App exists with OAuth2 token but token is expired/revoked:** `xurl auth status` shows an app with `oauth2: YOUR_USERNAME`, but `xurl whoami` returns 401 Unauthorized.

Verify with:
```bash
xurl whoami
```
If this returns a 401 error despite `auth status` showing a token, the token has expired or been revoked.

- **DO NOT attempt to post** — every post call will 401.
- Log the failure to `post_log.json` with status `FAILED` and error `xurl OAuth2 token expired/revoked (401)`.
- Update the block reason from the stale "no apps registered" diagnosis that earlier runs may have set.
- **Still proceed with Step 1 (load/generate content) and Step 2 (validate)** — same as State A: content is ready-and-waiting. Only skip posting steps (Steps 3-6). Then go to Step 7.
- **Increment `days_blocked` in post_log.json** — track consecutive blocked days.

State B fix (re-auth — user must complete outside agent):
```bash
xurl auth oauth2 --app default YOUR_USERNAME
```
This opens a browser for OAuth2 re-grant. No need to re-register the app — just refresh the token.

---

**State C — Fully authenticated:** `xurl auth status` shows an app with an active token AND `xurl whoami` returns the user profile.
- Proceed to Step 1.

### Step 1: Load Today's Content
Read from: `~/NurseRob_PeptideEmpire/content/YYYY-MM-DD_posts.md`
If file doesn't exist → call `peptide_content_operator` to generate it.

**Freshness check — content file may have been regenerated since last log entry:**
If a post_log entry for today already exists, compare the content file's modification time (`stat -c %Y content_file`) against the log entry's `last_attempt` timestamp. If the content file is newer, the file was regenerated after the log was written — the log contains stale titles, tweet counts, and validation details.
- Set a flag `content_regenerated=true` for this run.
- **Re-read the content file from scratch** — do NOT reuse cached titles or tweet counts from the log.
- **Re-validate all posts** even if the log says they passed (the content may be completely different).
- **Overwrite the stale log entry** with fresh titles, tweet counts, validation results, and `last_attempt` timestamp.
- **Update the dashboard** with the corrected content info (titles, slot descriptions, post count).

📦 Reference: [`references/content-freshness-check.md`](references/content-freshness-check.md) — detection pattern, tweet-count metadata drift, and post-regeneration parsing rules.

#### Step 1A: Detect Branded Images (Manual Attach)

Images are auto-generated by `peptide_content_operator` Step 5 and saved to Desktop. On X API free tier, media_ids are rejected on tweet creation — images cannot be attached via API. They're ready on Desktop for manual attach in the X app.

ls "/mnt/c/Users/Robert/Desktop/Daily Brief/NurseRob_PeptideEmpire/images/YYYY-MM-DD_post1.jpg" 2>/dev/null && echo "POST1_IMAGE_READY" || echo "POST1_NO_IMAGE"

```

If images found, note them in the Discord cross-post: "🖼️ Images ready on Desktop for manual attach."

**Upgrade path:** X API Basic ($100/mo) enables media_ids on tweet creation. At that point, add `--media-id` to xurl commands. Flag is `--media-id` (not `--media`).

### Step 2: Validate Each Post
**CRITICAL — Before validating, the `nurse-rob-x-reply-guidelines` skill (v1.3) MUST already be loaded.** The cron job loads it as the first skill. Apply the full 8-point Pre-Reply Checklist + Section 12 FDA/FTC language scan + Section 13 word boundary rules to EVERY post.

Check for:
- [ ] **8-point Pre-Reply Checklist (Section 11) — ALL pass**
- [ ] **Language Scan (Section 13) — no HIGH-RISK words, MEDIUM-RISK only in safe phrasings**
- [ ] **Word Boundary Rule applied — "health" ≠ "heal", "NurseRobHealth" is always safe**
- [ ] Disclaimer present (Section 2 mandatory boilerplate)
- [ ] CTA included (pushes to nurserobhealth.com/wolverine-stack)
- [ ] Under character limit (X: 280 per tweet, plus thread)

If validation fails: log issue, skip post, alert dashboard with 🔴.

### Step 3A: Pre-Post Check — Posts Already Live?
**Check if today's posts were already posted by an earlier process** (manual post, earlier cron run, or non-scheduled agent run that beat the scheduled cron).

Look for three conditions, in order:

**Condition 1 — Log says POSTED but DID NOT AGE-TEST THE CONTENT:**
```
post_log.json["YYYY-MM-DD"]["slots"]["morning"]["status"] == "POSTED"
```
AND freshness check passed (file mtime < log last_attempt — content was NOT regenerated after posting).

If both hold, the post log is trustworthy at face value. **Still verify on X** via Condition 2.

**Condition 2 — Verify on X via ID lookup (cheap, 1 request):**
```bash
# Use xurl get on the saved post_id — single request, no search cost
xurl get [POST_ID_FROM_LOG]
```
If the return code is 0 (exit code 0), the tweet exists. `xurl get` may return `{}` for some tweet types — in that case, fall back to a single search (not 3 retries):
```bash
xurl search "from:NurseRobHealth [first 10 words of opener]"
```

**Condition 3 — Verify thread replies via conversation_lookup (single request):**
```bash
# conversation_id lookup returns all replies — single request
xurl get "https://api.x.com/2/tweets/search/recent?query=conversation_id:[OPENER_ID]&tweet.fields=public_metrics" 2>/dev/null
```
Expected: the opener + all numbered replies. Count must match the log's `tweet_count`.

**If all three conditions pass (log trustworthy + post live on X + thread intact):**
- Posts were already made by an earlier process.
- Skip Steps 3B-6 (no re-posting, no re-logging).
- Still do Step 7 (queue next slot — check if evening slot also needs posting).
- Produce a brief "already posted" report with verification results.

**If Condition 1 fails (content regenerated):**
- Treat the log as stale (per freshness check in Step 1).
- Re-validate from scratch (Step 2).
- Determine whether the existing X post matches the new content or is outdated. If outdated, the user may need to delete the old post and re-post. Flag in the output.
- Fall through to Step 3B if the new content hasn't been posted yet.

**If Condition 2 or 3 fails (log says POSTED but X doesn't show it):**
- The log entry is corrupted/stale — the post was never actually made or was deleted.
### Step 3B: Post to X via xurl CLI (Text-Only — Free Tier)

**⚠️ X API Free Tier:** Media upload works but tweet creation rejects `media_ids` (verified May 17, 2026). Images auto-generate to Desktop for manual attach. Post text-only via API.

**⚠️ Quoting warning:** Many tweet texts contain apostrophes and single quotes. Use `execute_code` with Python `subprocess` to avoid shell quoting issues.

**For threads** (multiple tweets with delays):

```python
import subprocess, json, time

def run(cmd, timeout=30):
    r = subprocess.run(cmd, capture_output=True, text=True, timeout=timeout)
    if r.returncode != 0:
        print(f"FAILED: {r.stderr}")
    return r.stdout

# Post opener — text only (free tier)
opener_text = "🧵 [thread opener text]"
result = json.loads(run(["xurl", "post", opener_text]))
opener_id = result["data"]["id"]
thread_url = f"https://x.com/NurseRobHealth/status/{opener_id}"

# Reply all numbered tweets to OPENER_ID
replies = ["1/ ...", "2/ ...", ...]
for reply in replies:
    time.sleep(10)
    r = json.loads(run(["xurl", "reply", opener_id, reply]))

print(f"THREAD_URL: {thread_url}")
```

**For single posts:**
```bash
xurl post "[content]"
```

**Upgrade path:** X API Basic ($100/mo) enables media_ids. Flag is `--media-id` (not `--media`). When upgraded, add to opener: `opener_cmd.extend(["--media-id", media_id])`.
xurl post "[content]" --media-id "12345"   # --media-id, NOT --media
```

⚠️ **X API FREE TIER LIMITATION:** Media upload (`xurl media upload`) succeeds and returns a valid media ID, but attaching media to tweets via `--media-id` or the raw API `media: {media_ids: [...]}` field fails with "Your media IDs are invalid." This is an X platform restriction — only Basic ($100/mo), Pro ($5,000/mo), and Enterprise tiers support media attachments. On free tier, all posts go out text-only. Images are still generated and saved to Desktop for manual attachment from the X app. Do NOT attempt media attachment unless `xurl whoami` confirms a paid API tier.

**Poll — xurl has NO native poll command, use raw API**
Poll duration_minutes: 60|240|420|720|1440 (1hr/4hr/7hr/12hr/24hr)
Each option must be 1-25 chars, 2-4 options required
```bash
xurl -X POST /2/tweets -d '{"text":"[poll question with disclaimer + CTA]","poll":{"options":["Option A","Option B","Option C","Option D"],"duration_minutes":1440}}'
```

### Step 5: Log Post
Save to: `~/NurseRob_PeptideEmpire/content/post_log.json`

> **Real format**: The log is a date-keyed dict (not a flat array).
> See `references/post_log-structure.md` for the full schema with exact field names, types, and nesting.

**Verification** — `xurl post` returns a JSON with the post `id` and text on success. That IS verification — the tweet exists. No search needed.

**For threads**, verify reply count matches with a single conversation lookup:
```bash

### Step 7: Queue Next Slot
Set status for next slot. If next slot has content, confirm ready. If not, flag for content generation.

## Content Queue Logic
- Content is pre-generated to date-stamped files
- Scheduler reads file → posts → logs → updates dashboard
- Missing content file → call peptide_content_operator
- <2 posts in file → post what's available, flag gap
- Never double-post (check post_log.json for existing URLs)

## Pitfalls
- **`xurl post --media` vs `xurl reply --media-id` — different flags**: `xurl post` attaches images via `--media PATH` (file path). `xurl reply` attaches images via `--media-id MEDIA_ID` (a media ID string from a prior `xurl media upload` call). The scheduler only attaches images to the opener (`xurl post --media`) — replies are text-only. If the design ever changes to put images on specific reply tweets, the two-step upload-then-reply flow is required: `xurl media upload path/to/image.jpg` → captures media ID → `xurl reply TWEET_ID "text" --media-id "media_id_string"`.
- **Content file regenerated between scheduler runs**: The content file (`YYYY-MM-DD_posts.md`) can be updated/replaced by `peptide_content_operator`, manual edits, or another cron job **after** the initial post_log.json entry was written. This causes the log to contain stale titles, wrong tweet counts, and outdated validation results. Always do a freshness check (Step 1) by comparing content file mtime against the log entry's timestamp. If the file is newer, the entire log entry for today is untrustworthy — re-read and re-validate everything from scratch.
- **xurl auth missing (State A)**: First-run failure. `xurl auth status` shows "No apps registered" → every post call will 401. Block posting entirely until user completes one-time OAuth setup. Do NOT try to work around this — there is no fallback path.
- **xurl token expired (State B)**: App exists with OAuth2 token showing a username, but `xurl whoami` returns 401. The token was either revoked at the X developer portal or expired. **Do NOT re-run `xurl auth apps add`** — the app is already registered. Just run `xurl auth oauth2 --app default NurseRobHealth` for a fresh OAuth2 grant. This state often follows State A after the user partially sets up xurl but the token expires before first use. Diagnose with `xurl whoami` — do NOT rely solely on `xurl auth status` to confirm working auth.
- **Search verification race condition**: Immediately after posting, `xurl search` may return 0 results even for a successfully published tweet, because the X API search index takes 30–60 seconds to reflect new tweets. This is NOT a sign of a failed post — the `xurl post` call returned a valid `post_id`. Always wait before verifying, and escalate query specificity (use an 8–15 word unique phrase, not 4–5 common words) before concluding the tweet is missing. Observed May 16, 2026: search with "hope gap nobody talks about" (5 words) → 0 results; search with "sometimes the peptide just doesn't work" (8 words, more unique) after 30s → found the tweet.
- **`xurl get` false negatives — search, don't just fetch**: When verifying whether a tweet from the log actually exists, `xurl get <id>` may return `{}` with "Error: request failed" even for valid, live tweets. This was observed on May 14, 2026: tweet `2054925681067164079` returned empty via `xurl get` but was confirmed live via `xurl search`. **Always use `xurl search` to verify tweet existence**, not `xurl get`. If `xurl get` returns empty, do NOT conclude the post was never made — run a search before taking corrective action. `xurl get` may only support certain tweet types (e.g., quotes, threads with replies); for single posts or Q&A tweets, search is the only reliable verification path.
- **Stale auth diagnosis in logs masks working credentials (State D — false block)**: post_log.json and `dashboard/metrics.json` may say "BLOCKED — xurl 401" because a non-default app (e.g., `default (no credentials)`) has an expired token, while the actual default app (marked with `▸`, e.g., `nurse-rob`) is fully authenticated and working. This happens when an earlier setup created a `default` app that expired, then a later setup created a working `nurse-rob` app and set it as default — but the logs were never updated. **Always trust live `xurl whoami` over any stale log entry.** If `xurl auth status` shows a default app (▸) with `oauth2: NurseRobHealth` and `xurl whoami` returns 200, post_log.json saying "BLOCKED" is stale — proceed to post. Fix the log entry with the real auth status after determining which app is actually the default.
- **Intra-day stale slot status in post_log.json**: A morning scheduler run may set the evening slot's `status` to "READY — blocked by auth" because at that point auth was broken. Later, if auth is fixed and the evening scheduler runs, **do NOT trust the old slot status** — it's stale from earlier in the same day. Re-check `xurl whoami` fresh and overwrite the slot entry entirely (status → POSTED, add post_id/url/tweet_count/posted_at) rather than trying to patch around the old status.
- **Post-block dashboard cleanup required**: When posting succeeds after a period of being blocked (auth fixed), the stale dashboard fields must ALL be cleared. **The canonical path is `~/NurseRob_PeptideEmpire/dashboard/metrics.json`** (not `content/metrics.json` — that path may not exist):
  - `dashboard/metrics.json`: `content.block_reason`, `content.day_of_block`, `content.blocker`, `content.setup_command`, `content.posting_status` (change from BLOCKED to POSTED/PARTIAL), `content.posts_today`, `content.posts_posted`
  - `dashboard/metrics.json.degraded_mode`: clear `xurl`, `fix_primary`, `impact` — update to reflect OK status
  - `dashboard/metrics.json._attention`: rewrite to reflect current status (not stale BLOCKED)
  - `dashboard/metrics.json.alerts.critical`: remove any `xurl OAuth2 TOKEN EXPIRED` entries
  - `dashboard/metrics.json.alerts.good`: add success entry
  - `dashboard/metrics.json.degraded_note`, `dashboard/metrics.json.degraded_fix`: update or clear
  - `dashboard/metrics.json.leads.xurl_authenticated`, `dashboard/metrics.json.leads.lead_sniper.xurl_authenticated`: set True
  - `dashboard/metrics.json.days_blocked`: reset to 0
  - `post_log.json.days_blocked`: reset to 0
  - `post_log.json.last_error`: clear
  - `post_log.json['YYYY-MM-DD'].status`: change from `CONTENT_READY_AUTH_BLOCKED` to `POSTED_MORNING_READY_EVENING` (or appropriate)
  - **`post_log.json` top-level metadata ALSO stale**: update `log['status']` (e.g. from `"BLOCKED — xurl 401"` to `"POSTING_OK — ..."`), `log['error']` (clear or set to resolved), `log['fix']` (set to None). The update_post_log.py script does NOT touch these — they must be swept manually with execute_code + subprocess raw I/O.
  - **`post_log.json` top-level `posts[]` array**: sweep any "blocked by auth" text from each entry's `status` field. Example: change `"READY — blocked by auth"` to `"READY (from previous batch)"`.
  - Any `lead_followup.status` or `lead_followup.note` referencing the old block
- **Day counter tracking**: Keep a `days_blocked` field in `post_log.json`. Increment each day auth is broken, reset only when a post succeeds. This gives a clear picture of lost content days.
  - ⚠️ **update_post_log.py's `compute_summary` does NOT implement this correctly**: It counts ALL historical date entries with BLOCKED/401/FAILED auth_status, not just consecutive recent ones. After posting resumes, old blocked days from weeks ago still count. Manually zero `log['days_blocked'] = 0` in execute_code after any successful post.
- **JSON file I/O in execute_code sandbox**: When modifying post_log.json, metrics.json, or any other JSON file from within `execute_code`, **do NOT use `read_file`/`write_file` hermes_tools** — they return line-numbered content (`"     1|{..."`) that breaks `json.loads()`. Instead:
  - **Read**: `json.loads(subprocess.run(["cat", path], capture_output=True, text=True, timeout=5).stdout)`
  - **Write**: `subprocess.run(["tee", path], input=json.dumps(data, indent=2), text=True, timeout=5)`
  - **Read (alternative)**: `json.load(open(path))` also works inside execute_code
  - This applies to ALL JSON file operations against post_log.json, `dashboard/metrics.json`, lead_log.json, etc. When the pitfalls reference `metrics.json`, the canonical path is always `~/NurseRob_PeptideEmpire/dashboard/metrics.json`.
- **Do NOT use `xurl search` for routine post verification** — it costs a Read request per call and the search index has a 30-60 second propagation delay. Trust the `xurl post` response (returns post_id on success). Use `xurl get [post_id]` for cheap verification. Only fall back to search if `xurl get` returns empty AND there's reason to suspect the post failed. This change saves ~$0.10-0.15/day in API costs (May 17, 2026 optimization).
- **Lead sniper frequency drives Read request costs.** At 4x/day, ~$0.25-0.30/day in search requests. At 2x/day, ~$0.12-0.15/day. Tune based on lead volume. Morning + midday is the sweet spot for Nurse Rob's current traffic level.
- Timezone: ALWAYS use MST (America/Phoenix — Mesa, AZ)
- X rate limits: max 50 posts/day, space them out
- Thread posts: post sequentially with 10-sec delay between tweets
- Don't post if validation fails — skip rather than post bad content
- Monitor for shadow-ban patterns (engagement suddenly drops)
- xurl installed but unauthenticated: `which xurl` succeeds but `xurl auth status` shows no apps → same as above, block and surface setup instructions
